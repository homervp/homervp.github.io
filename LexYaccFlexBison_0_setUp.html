Lex, Yacc, Flex and Bison: intro and setting up on Windows:

First off, Lex vs Flex and Yacc vs Bison. Can you see the similarity? The F in Flex stands for Fast. Lex on its own stands for Lexer. Yacc does not stand for anything in particular, but if you realise that there's an animal out there called a yacc, you'll see that Bison does have some similarity (they are pretty much the same species). Those last two are Parsers. Or rather they generate parsers. The second pair is an "evolution" of the first, but the syntax of the first pair is mostly the same as that of the second, in fact I completed a project in Flex and Bison by reading a book on Lex and Yacc (Lex & Yaxx, John R. Levine, Tony Mason, Doug Brown, O'reilly publishing), which did reference the differences between the two pairs, so no, the first are not outdated, rather a matter of taste.

We use these sets of tools to create the foundations of a compiler. A lexer is a component that detects individual words and tokenizes them, that is, it turns them into tokes, which are pretty much just numbers that indicate their function. I won't get into the details, but you will pretty much be #defining constants for the different words to give.

A parser, or syntax analyzer, is a tool that takes tokens and, by observing the order in which they come, figures out how they're supposed to interract. And no, figures out is not too innacurate, because you can make a language with ambiguity, in which case you pray the parser will figure out what you were trying to do.

So, what will we accomplish? We won't be making an entire compiler in this series, we will only recognize what is being done. Isn't that all we need? No. Compilers substitute code with other code. So before you make a compiler, you'll have to read up some assembly for the system you use (at the time of this writing, some windows assembly is planned but not yet uploaded). You could, however, try to compile from your language to C, which should be easy enough..
